# 通用文档写作与排版规范

## 一、适用原则

1. 本规范用于约束各类项目文档的写作、排版、交付和风格统一。
2. 通用要求优先执行；具体文档类型的特殊要求作为补充执行。
3. 文档生成时，先套用本规范，再按主题识别补充对应类型要求。

## 二、核心目标

1. 排版稳定，适合阅读、共享、打印和转为 `Word`、`PPT`。
2. 语言自然、直接、正式，尽量去除明显的人工智能写作痕迹。
3. 结构清晰，层级分明，重点突出。
4. 信息可核查，来源可追溯。

## 三、去 AI 化要求

### 1. 语言风格

1. 使用正式、直接、自然的中文表达。
2. 优先使用常见书面语，不堆砌修辞，不故作高深。
3. 句子尽量写实，少写空话、套话、姿态性语言，做到“说人话”，但不流于口语化。
4. 一段只表达一个主要意思，避免连续堆叠判断。
5. 结论要明确，但不能悬空，后面要跟说明或依据。
6. 涉及科研经历、实践案例、项目故事时，不能只写一句话式例子，必须交代背景、过程、判断依据和结果。

### 2. 应避免的表达

1. 避免明显“人工智能腔”词语，例如：
   - 张力
   - 断点
   - 参差
   - 赋能
   - 收口
   - 抓手（除非场景确实合适）
   - 闭环（可用，但不要反复滥用）
   - 口径（如非统计、抽样、财务、核算等必要场景，尽量不用）
   - 颗粒度（非必要不用）
   - 主线（如能换成“重点问题”“核心问题”“主要线索”等更具体写法，优先替换）
   - 高压版、基础版、进阶版、极限版（在备课和问答材料里尽量不用）
2. 避免自我说明式废话，例如：
   - 本文将……
   - 以下内容可用于……
   - 如果从某种角度来看……
   - 可以理解为……
   - 不难发现……
   - 值得注意的是……
3. 避免过强的对话感和口语感，例如：
   - 可以这样讲
   - 跟谁讲
   - 如果你愿意
   - 对老板来说
   - 更像是……
4. 避免机械对仗句式反复出现，例如：
   - 不是A，而是B
   - 既……又……还……
   - 一方面……另一方面……
5. 避免“先双重否定，再转肯定”的表达习惯，例如：
   - 这些群体的学习投入并不低，主要困难在于……
   - 该问题并非来自学生意愿，而是来自……
   这类句子容易显得拧巴，建议直接写成肯定句：学习投入总体正常，主要困难是……
6. 避免泛泛判断，不写没有承接内容的“高度概括句”。
7. 非必要不要频繁使用破折号、双引号、书名号套引号、连续括号说明等符号堆叠。
8. 符号应服务于表达清晰，不应制造“像机器整理出来的文字”那种观感。
9. 特别是以下情况应尽量避免：
   - 一句话中多次使用破折号
   - 大量使用双引号强调普通词语
   - 连续出现括号补充说明，打断阅读节奏
   - 用符号代替正常句子衔接
10. 避免把小标题写成机械标签样式，例如：
   - 政策依据。
   - 设题动因。
   - 主要关注点。
   这类写法容易带出明显整理痕迹，正式材料中应改为不带句号的短标题或直接并入正文。
11. 避免把小标题写成明显的模板整理腔，例如：
   - 本段目标
   - 主问池
   - 追问池
   - 老师抓手
   - 本次重点
   这类标题虽然整齐，但很容易出现机器整理感。优先改成更自然的写法，例如：
   - 这一段练什么
   - 可以这样问
   - 继续追问
   - 上课时重点看
   - 这次主要改什么
12. 避免“标准答案腔”：
   - 每段都很完整，但没有轻重
   - 句子太平，像工作汇报
   - 抽象判断太多，具体场景太少
   - 满篇都是能成立但不好用的套话
13. 面向非心理/统计读者的正文，避免直接使用分析术语堆叠，例如：
   - 关键变量联动机制
   - 结果显示（后面不接具体事实时）
   - 回归结果表明、显著预测、模型解释率（这类写法可放方法或附录）
   正文建议改成：关键因素之间的关系、主要影响因素、主要趋势、主要发现等自然表述。
14. 避免“写作腔动词”连用，例如：
   - 讲透、练透、打透、拉满
   这类词在报告里容易显得虚。建议改成具体动作，如“说明清楚”“训练到位”“把流程做实”。
15. 避免标题或小标题重复同义说明，例如：
   - 原因分析：为什么会出现这些问题
   - 政策依据：相关政策依据
   建议只保留一个表达，标题要短、直接、信息不重复。
16. 面向领导阅读的正文，避免偏技术化词语，例如：
   - 中间区间、转化效应、机制变量、效应路径
   建议改成：中等水平、后续变化、关键因素、影响过程。
17. 专题报告正文尽量少堆统计术语和数值细节。
   - 正文以判断和问题为主，数据用于支撑，不作为叙述主体。
   - 详细统计值优先放附表、脚注或括注，不在正文连续罗列。
18. 避免“先否定再转折”的写法（包括双重否定和硬转折）：
   - 例如“并不低……主要是……”“不是……而是……”
   建议改成直接句：先讲事实，再讲原因，减少拧巴表达。
19. 避免用空泛程度词起句或作判断主干，例如：
   - 影响很直接
   - 差异很明显
   - 仅有……还不够
   这类写法信息量低、语气生硬，建议改成事实句：先写“具体表现是什么”，再写“对工作意味着什么”。
20. 避免连续排比提问句直接入正文，例如：
   - 哪些会变、哪些会被影响、还要补什么
   这类写法在报告里容易出现“提纲感”。建议改成自然叙述句，不要把提问清单直接当正文。
21. 避免“变量口吻”或“分析口吻”直接做段首判断，例如：
   - 某类差异已经出现，且与某因素密切相关
   建议改成“主体 + 情境 + 表现”写法，先写谁在什么情境下出现了什么变化。
22. 现状部分段首避免抽象标签式开头，例如：
   - 某项工作很重要
   - 某类差异明显
   建议改成“先交代对象，再交代变化”，用完整叙述句开段。
23. 避免模板化转场句反复出现，例如：
   - 从政策层面看
   - 从实践层面看
   - 从数据层面看
   这类句子容易像“答题框架”。建议改成情境承接句，例如“随着……”“在……背景下……”“结合……变化……”。
24. 避免“`不够 + 单字形容词`”的省略写法，例如：
   - 不够紧、不够稳
   这类写法口语感重、表达含混。建议改成完整书面表达：不够紧密、不够稳定、衔接不足、连续性不足。
25. 避免口头语或指令式短句直接作为报告正文，例如：
   - 把这项工作前移
   - 要持续跟进
   这类句子常见于讨论和会场发言，不适合作为正式书面报告正文。建议改成完整书面句：该项工作应前置实施；应建立持续跟进机制。

### 3. 推荐的表达方式

1. 先写判断，再写解释。
2. 先写事实，再写含义。
3. 先写问题，再写原因和建议。
4. 能具体就不抽象，能落地就不空泛。
5. 多用简单词，少用花词。
6. 能把过程讲清楚时，不要只写结论。
7. 能展开关键假设、方案比较和结果可能性时，不要压缩成口号式表达。
8. 对执行建议、变量说明、分析限制等内容，优先使用正常书面表达，不用项目管理腔或分析腔词语硬做概括。
9. 备课、模拟面试、问答题库类文档，应尽量写成老师上课时真会说的话，不要写成目录说明书。
10. 简历、研究计划、案例提问类材料，应优先把问题写具体，把追问写具体，把修改点写具体。
11. 现状描述优先使用“对象 + 情境 + 变化”句式，少用标签判断句开头。

### 4. 面试与备课文档补充要求

1. 这类文档优先使用直接、朴素、贴场景的表达。
2. 标题尽量短，少用模板化标签。
3. 少写“形成主线”“守住说法”“提高张力”这类抽象话，多写：
   - 先把研究问题说清
   - 前后不要改变量
   - 回答不要绕
   - 先讲结论，再讲例子
4. 轮次区分优先写“第一次、第二次、第三次”，或“基础练习、研究计划加压、正式模拟”，尽量不用“高压版、进阶版、极限版”。
5. 提问要贴简历和计划书原内容，少出放之四海皆准的空题。
6. 回答点评要尽量落到具体经历、具体判断、具体做法、具体修改，不要只写空泛评价。

## 四、排版样式要求

### 1. 标题层级

1. 一级标题、二级标题、三级标题必须有明确层级差异。
2. 不能只靠全部加粗区分层级，还要通过字号、段前段后距体现层次。
3. 大模块之间要有明显间隔，必要时使用分页符。
4. 小模块之间至少留出适当空白，避免上下粘连。
5. 如使用条目式小标题或段内小标题，默认不在标题词后加句号。
6. 如使用“政策依据”“主要问题”“工作建议”“对应引用”等标签式小标题，宜直接写为无标点短标题，后接正文或列表，不写成“政策依据。”这类样式。

### 2. 正文排版

1. 中文正文采用正式中文材料习惯，段落整齐，视觉密度适中。
2. 正文中文应有首行缩进。
3. 行距以 `1.5` 倍为主，兼顾阅读和页数控制。
4. 段前段后距应控制在适中范围，避免一页没几个字就翻页。
5. 不要把所有内容拆成过多短段，否则文档会显得很散。
6. 也不要把大量内容写成超长段，否则阅读负担重。

### 3. 字体与字号（Word 交付默认标准）

1. 中文正文字体：宋体。
2. 英文正文字体：`Times New Roman`。
3. 正文字号：四号。
4. 如无专项要求，文档主标题默认使用二号黑体，居中排布，颜色为黑色。
5. 标题字号应高于正文，并按层级递减。
6. 重点句可加粗，但正文大面积加粗会破坏层次，应严格控制。

### 3A. 中文 Word 基础版式刚性要求

1. 中文正文段落应设置首行缩进，默认按两个汉字字符处理。
2. 正文默认使用 `1.5` 倍行距，段前段后一般设为 `0` 或适中固定值，不得造成段落忽松忽紧。
3. 正文默认两端对齐，避免左侧整齐、右侧参差明显的网页式排版观感。
4. 标题必须显式设置为居中，不能依赖编辑器或模板默认样式。
5. 标题、正文、编号如无特别要求，一律使用黑色字体，不使用 `Word` 主题蓝、自动强调色或超链接蓝作为正文显示色。
6. 中文字体、字号、缩进、对齐方式应直接写入 `Word` 样式或段落属性，不能只在中间格式中“看起来像是对的”。

### 3B. Word 生成与转换约束

1. 交付目标如果是 `docx`，优先使用可直接生成 `Word` 结构的方案，例如 `python-docx` 或带参考模板的稳定生成链路。
2. 不应把 `RTF -> docx` 作为默认交付路径，尤其在中文材料场景下，除非已经验证编码、字体和样式完全稳定。
3. 涉及中文内容时，生成流程必须显式保证 `UTF-8` 编码，避免出现转码后乱码。
4. 文档生成后，必须做一次结果核查，至少检查四项：中文是否乱码、标题是否居中、标题是否为黑色、正文是否首行缩进。
5. 如果通过中间格式转换生成 `docx`，应把“内容正确”和“样式正确”分开核查，不能只检查文本是否存在。
6. 需要长期复用的文档生成任务，应优先沉淀为脚本化模板，避免每次依赖临时转换和手工修补。
7. 需要生成目录时，章节标题必须绑定 `Word` 内置标题样式（`Heading 1/2/3` 或对应本地化样式），不能只靠手动加粗或字号模拟标题。
8. 目录交付前必须核查两项：目录字段可更新；更新后能正确抓取各级标题并生成页码。
9. 需要“可跳转目录”时，目录条目必须支持内部锚点跳转（`Ctrl + 点击` 可定位到对应章节），不能只做纯文本目录。

### 4. 重点展示

1. 关键判断句、转折句、结论句可以加粗。
2. 加粗应服务于阅读，不应把整页做成“满页重点”。
3. 一般每个一级部分控制在 `1-3` 处重点即可。
4. 编号、字段名、结论标签可以加粗；正文解释一般不加粗。

### 5. 列表与表格

1. 需要分点说明时，优先使用编号列表或项目符号。
2. 项目并列时，用列表；连续论证时，用正文段落。
3. 不要把适合列点的内容硬写成一大段。
4. 不要用大表格承载大段正文内容。
5. `Markdown` 里的竖线表格不应机械转成大面积 `Word` 表格。
6. 如内容偏叙述型，优先改成“条目式正文”。
7. 只有字段短、内容整齐、行列关系明确时，才适合用小表格。

### 6. 链接与目录

1. 参考网址应处理为真正可点击的超链接。
2. 目录应使用正式目录字段，支持跳转导航。
3. 目录不是装饰，必须和标题层级对应。
4. 文档尾部如有附录、参考链接、核查链接，应保留统一样式。
5. 如文档属于正式中文材料，超链接默认不保留蓝色强调样式；如需保留可点击性，应同时控制其显示样式与正文一致或整体协调。
6. 目录的前提是“样式层级”而不是“视觉层级”：只有被设置为 `Heading` 的标题才能被目录识别。
7. 导航体验至少满足其一：目录字段更新后可跳转；或目录条目已写入内部锚点，可直接 `Ctrl + 点击` 跳转。

## 五、术语与名称写法

1. 机构、组织、计划、项目、数据库等名称首次出现时，统一写为：`缩写（英文全称，中文全称）`。
2. 后文再次出现时，可只写缩写或规范简称。
3. 英文缩写首次出现不能裸写。
4. 同一文档中，术语写法必须统一，不能前后混用。
5. 常见研究术语是否解释，应看文档对象；如无特殊要求，不必逐一做百科式解释。

## 六、引用与核查要求

1. 只要引用了政策、研究、案例、试点、官网信息、机构资料，原则上都应保留核查依据。
2. 引文不宜大段复制原文，优先写“原文核心内容（转述）”。
3. 文末应设置“引用依据”“参考资料”或“核查链接”部分，便于回查。
4. 如果某条判断来自多个来源，应尽量保留主来源和交叉来源。
5. 如果某条内容属于推断或归纳，应与事实性表述区分开。

## 七、通用结构建议

1. 大多数文档都应有清晰主线，避免堆材料。
2. 推荐使用以下基础结构之一：
   - 背景 -> 问题 -> 分析 -> 建议 -> 结论
   - 目标 -> 现状 -> 缺口 -> 方案 -> 路径 -> 成果
   - 证据 -> 判断 -> 方案 -> 风险 -> 推进建议
3. 文档不是目录堆砌，章节之间要能接上。
4. 每个一级部分最好能回答一个明确问题。
5. 每一部分结束时，宜给出一两句收束判断，便于后续汇报和转为 `PPT`。
6. 如材料属于完整文章、策划案、研究方案或项目论证，不得只写“观点 + 简例 + 结论”的缩略版，必须形成完整论证链。
7. 完整论证链至少应覆盖以下要素中的大部分：问题背景、核心假设、分析过程、方案设计、执行流程、结果可能性、风险点、讨论与调整方向。
8. 凡是用户明确要求“完整写”“讲透”“保姆级讲清楚”，应优先采用展开式写法，不得擅自压缩成提纲、摘要或短评。

## 八、可作为主题识别后的补充要求

以下内容不一定每次都启用，但在识别出对应文档类型后，应作为补充标准执行。

### 1. 科研综述类文档

特殊要求：

1. 不能只列文献题目，必须概括研究问题、方法、主要发现和局限。
2. 国内外文献应适度区分，不机械平均。
3. 如涉及 `Meta` 分析、系统综述、队列研究、横断面研究，应区分证据层级。
4. 研究结论与作者判断要分开写。
5. 文末应保留文献来源、数据库链接、编号或正式引用信息。
6. 如文中写到科研案例、研究故事、研究设想，不得只写“做了什么、效果很好”这一类空泛句，必须写清楚研究起点、研究设计、关键假设、实施过程、卡点、调整和结果解释。
7. 科研写作应尽量还原真实研究流程，让读者看得见“怎么得出这个判断”，而不是只看到结论。

### 2. 开题报告或研究方案类文档

特殊要求：

1. 研究背景要具体，不写空泛宏大叙事。
2. 研究问题、研究目标、研究意义应彼此区分。
3. 研究设计要写清楚对象、方法、变量、流程和预期结果。
4. 技术路线或研究路线应能直接画图或转成流程图。
5. 如涉及问卷、量表、访谈、实验设计，应说明工具来源和适用性。
6. 必须明确写出研究假设或核心判断，不能只写研究意向。
7. 必须把研究流程拆开说明，至少包括：前期准备、实施步骤、数据获取、分析方法、结果解释和可能调整。
8. 必须说明结果可能出现的几种情况，以及每种情况下如何解释、如何处理。
9. 如为完整方案稿，应增加讨论部分，说明潜在偏差、局限、备选方案和优化空间。

### 3. 政策研究或决策咨询类文档

特殊要求：

1. 政策依据要写具体，不能只写“已有政策支持”。
2. 现实问题要落到执行层，不停留在原则表述。
3. 建议部分应区分：短期可做、中期推进、长期机制。
4. 对上汇报材料要突出“为什么现在做、从哪里切、能形成什么结果”。
5. 如引用地方经验，应说明其可复制性和局限。
6. 对策方案不能只列口号和动作清单，应写清楚提出该方案的逻辑、实施条件、推进流程、预期结果和风险点。
7. 如属策划案、行动方案、专题方案，应尽量形成完整文章结构，至少交代问题分析、方案选择依据、实施步骤、可能结果和讨论判断。

### 4. 合作论证或项目建议类文档

特殊要求：

1. 必须讲清楚合作双方各自缺什么、各自能提供什么。
2. 必须写明分工，而不是只写合作愿景。
3. 合作切口要具体，能落到项目、试点、培训、工具或评估机制。
4. 如提出新方案，应说明依据来自哪里，哪些是借鉴，哪些是新设计。
5. 最终应能沉淀为可汇报、可签报、可转为 `PPT` 的结构。
6. 方案部分必须写完整，不能停留在“建议开展某项目”这类一句话判断。
7. 应按流程展开说明：为什么这样设计、先做什么、后做什么、每一步由谁承担、可能出现什么结果、如果不顺利如何调整。

### 5. 资料汇编或证据库类文档

特殊要求：

1. 每条资料都应有明确分类。
2. 每条资料至少包含：主题、核心内容、用途或启发、来源。
3. 不能只登记网址和标题。
4. 政策、观点、实践、研究、问题、合作建议等应分组整理。
5. 如属借鉴类资料，应标明是“借鉴”“放大”还是“新提出”。

### 6. 会议纪要或访谈整理类文档

特殊要求：

1. 纪要应按议题整理，不要逐句堆录音内容。
2. 发言要点、分歧点、结论、待办事项应分开写。
3. 如用于后续推进，必须单列行动项、责任人、时间点。
4. 对口语化原话要做书面整理，但不能改变原意。
5. 重要表态或明确判断可保留原话风格，但应标清来源对象。

### 7. 课题申报或项目申报类文档

特殊要求：

1. 目标、内容、创新点、可行性、预期成果要分别展开。
2. 创新点不能写成口号，必须对照现有工作说明新增之处。
3. 项目基础、团队基础、实施条件要有具体支撑。
4. 预算、周期、阶段成果应与任务安排对应。
5. 文字风格要稳，避免宣传稿口吻。

### 8. 培训材料或课程讲义类文档

特殊要求：

1. 结构应以“对象是谁、要解决什么问题”为中心。
2. 每部分内容应可讲、可练、可落地，不能只有概念。
3. 如面向一线人员，应尽量加入场景、案例、操作提示。
4. 术语应做必要统一说明，避免听众看不懂。
5. 后续如要转 `PPT`，每部分应能拆成独立页面。

## 九、文件管理要求

1. 文档应分类存放，避免所有版本混在一起。
2. 需求说明、正式文稿、交付成品、脚本、临时文件应分目录管理。
3. 文件名应尽量包含主题、版本或日期，便于检索。
4. 可交付版本与工作草稿应分开保存。

## 十、实际执行建议

后续生成任意项目文档时，可按以下顺序执行：

1. 先套用本规范中的“去 AI 化要求”和“排版样式要求”。
2. 再判断文档属于哪一种类型。
3. 根据识别出的类型，补充对应“特殊要求”。
4. 最后检查三件事：
   - 语言是否自然、正式、无明显人工智能腔
   - 结构是否清楚、重点是否突出
   - 引用是否可核查、排版是否可直接交付

如用户要求写“完整文章”“完整策划案”“完整方案”，再额外检查四件事：

1. 是否把问题背景和核心假设写清楚了。
2. 是否把分析过程和执行流程讲清楚了。
3. 是否写出了结果可能性、风险点和讨论，而不只是单一路径。
4. 是否避免了“一句话故事”“一句话案例”“一句话科研经历”。

如交付物为 `Word`，在上述基础上再补做一次最小排版验收：

1. 标题是否为二号黑体、居中、黑色。
2. 正文是否为四号宋体。
3. 正文是否首行缩进、`1.5` 倍行距、两端对齐。
4. 中文是否存在乱码、异常符号或转换丢字。

这份规范可作为后续所有项目文档生成、修改和排版的统一约束基础。
